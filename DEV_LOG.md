## 2026-05-23 | 技术债#1：模糊搜索重修

### 问题

`complete_task` / `delete_task` 用 `LIKE '%keyword%'` 匹配标题，用户说"完成数学作业"匹配不到"高等数学第一章作业"（0条），或者"删掉作业"匹配到 10 条全报错。

### 改了啥

**A. `task_repo.rs` — 新增 `search_tasks_scored`**
- 把 query 拆成词（空格/中文标点分隔）
- 每个词在标题/标签/描述/备注里查 LIKE
- 命中打分：标题=10，标签=5，描述=3，备注=2
- 按总分降序返回 `Vec<(Task, i32)>`

**B. `tools.rs` — 工具定义加筛选参数**
- `complete_task`：新增可选 `status`（默认只搜未完成）、`priority`
- `delete_task`：新增可选 `status`、`priority`

**C. `tool_executor.rs` — 智能匹配分支**
- 0 条 → 检查是不是已完成，分别提示
- 1 条 / 分数碾压（第1名 ≥ 第2名×2）→ 自动执行
- 2-5 条接近 → 返回格式化列表，让 AI 请用户选
- 5+ 条 → 展示前5条，提示更具体的关键词
- `search_tasks` 工具也改用加权搜索

### 设计决策

- **不引入 FTS5**：几百条任务不需要全文索引，分词+Rust打分足够
- **complete_task 暂时不接 XP**：`execute_tool` 拿的是 `&Connection`，`task_repo::complete_task` 需要 `&mut Connection`。留了 TODO，后续改
- **分数碾压阈值 2x**：第1名分数是第2名2倍以上视为"明显胜出"，直接自动选

## 2026-05-22 | 尘笺后端打通：MD文件 + 时间线 + 提灯的日记

### 做了什么

尘笺从静态 UI 变成了完整可用的日记系统。Rust 后端 + React 前端 + 三个新 UI 组件。

### 设计决策

* **日省 vs AI 尘笺**：本来有两个重叠功能（日省按钮生成评价 + 日晷打开 AI 日记），后来统一了——日省是**写**的入口，日晷是**读**的入口。点击日省 → 提灯从多维度写一段旁白 → 保存到 `YYYY-MM-DD-ai.md` → 日晷面板阅读
* **提灯的日记**：AI 没有自己的生活，所以不是"AI 的日记"，而是"AI 对你这一天的旁白"。像一个了解你的朋友说"今天看你状态不错，但下午好像有点焦虑"
* **文件 I/O 合并到 repo**：本来计划单独的 `journal\\\\\\\\\\\\\\\_fs.rs`，后来发现 frontmatter 必须和 DB 元数据保持同步，分开反而增加复杂度，直接合并到 `journal\\\\\\\\\\\\\\\_repo.rs`
* **AppDataState**：新增一个 managed state 存 app\_data\_dir 路径，避免每个 command 都传 app\_handle
* **自动保存**：1.5s debounce，用户不用想"要不要保存"这个事

### 翻车记录

* `\\\\\\\\\\\\\\\_db\\\\\\\\\\\\\\\_state` vs `db\\\\\\\\\\\\\\\_state`：`save\\\\\\\\\\\\\\\_ai\\\\\\\\\\\\\\\_diary` 里把参数改成了 `\\\\\\\\\\\\\\\_db\\\\\\\\\\\\\\\_state`（标记未使用），但函数体里又用了 `db\\\\\\\\\\\\\\\_state.conn`，cargo check 直接报错。两个函数的命名搞反了

### 文件结构

```
journals/YYYY/MM/
  2026-05-22.md        ← 用户日记（YAML frontmatter + 正文）
  2026-05-22-ai.md     ← 提灯的日记（独立文件，不污染用户日记）
```

## 2026-05-22~23 | 日历从零到完整：三个阶段一口气干完

### 做了什么

日历（时序）页面从一个只有日期栏的空壳，变成了功能完整的日程系统。三个阶段一口气做完：

**Phase 1 — 能用**
- 周视图：真实日期 + 时间轴 + 事件色块 + 当前时间红线
- 后端 CRUD + 范围查询 + rrule 展开（先做了 WEEKLY）
- 任务同步：tasks.scheduled_at 自动出现在日历，虚线边框只读
- 创建弹窗：标题 + 时间 + 分类

**Phase 2 — 好用**
- DAILY / MONTHLY 重复展开补全
- EventDetail 详情弹窗：查看/编辑/删除，含重复规则显示
- .ics 文件导入：前端解析 iCal + 后端批量写入 + UID 去重
- 月视图：7×6 网格，点击某天进日视图
- 全天事件渲染

**Phase 3 — 强大**
- 近期视图（原名议程）：从当前时刻起算，按日分组
- 日视图：单天详细时间轴 + 拖拽调整
- 拖拽调整时间：周视图 + 日视图，30 分钟吸附，实时时间指示线
- 系统通知：浏览器 Notification，事件前 10 分钟提醒
- 重复事件单次编辑：检测实例 ID，"只改这一次"（exdate + 新建独立事件）vs "改所有"

### 设计决策

* **夜间时间压缩**：0-7 点压缩到 10% 高度，7-24 点占 90%。用户说"我经常熬到 23-24 点"，所以 23-24 不压缩
* **重叠事件分栏**：只有真正重叠的事件才分栏，不重叠的保持全宽。贪心算法检测重叠组
* **近期视图从当前时刻起算**：一开始改成了凌晨，用户说"从当前时刻这个设计非常精妙"，改回来。语义是"接下来还有什么"，不是"今天都发生了什么"
* **重复事件实例 ID**：`{base_id}_{YYYY-MM-DD}` 格式，nanoid 不含下划线所以能可靠分割
* **单次编辑策略**：exdate 排除原日期 + 创建独立事件，不修改父事件的 rrule

### 翻车记录

* **WeekView 高度塌陷**：`flex-1` + `absolute` 子元素导致零高度，设了显式 `height: 800` 才解决
* **Rust 重复代码块**：编辑 schedule_repo.rs 后残留了重复的 expand_weekly，"unexpected closing delimiter"
* **Rust DateTime 类型不匹配**：`make_instance` 期望 `DateTime<Local>` 收到 `DateTime<FixedOffset>`，改成泛型 `Tz: chrono::TimeZone`
* **空白页面**：`onImportIcs={onImportIcs}` 写成了 `onImportIcs={handleImportIcs}` 变量名不对
* **月视图右翻页失灵**：`setWeekMonday(getWeekMonday(prev))` 把日期转成了周一，破坏了月份导航
* **拖拽与文字选择冲突**：桌面应用的 webview 里，浏览器原生文字选择会干扰自定义拖拽。`e.preventDefault()` + 全局 `user-select: none` 解决

### 额外收获

这次一起做的不只是日历：

* **布局重构**：页面从 `min-h-screen`（整页滚动）改成 `h-screen`（固定视口），只有内容区滚动。导航栏、控制区都固定
* **命名规范化**：用户反思后决定导航标签用直白名字（提灯→助手、尘事→任务、时序→日历、尘笺→日记），古风只用在装饰性文字。"UI 命名是功能语言，不是文学语言"
* **全局禁止文字选中**：`user-select: none` + 输入框例外，桌面应用应有的样子

- 44 个文件变更，5373 行新增
- 7 个 Tauri Commands
- 8 个 schedule 组件
- 4 种视图（周/日/月/近期）

## 2026-05-25 | 主题系统救火 + 自动保存 + 农历 + 小修小补

### 主题色硬编码大扫除

墙角梅主题（深红bg + 浅色card）暴露了大量硬编码问题。根本原因：很多组件用 `t.isDark`（页面背景明暗）来推导卡片上的颜色，但卡片本身可能是浅色。修了：

- **LanternModal（提灯弹窗）**：所有 `t.isDark` → `cardIsDark`（基于 `t.card` 计算），文字从 `t.text` → `t.cardText`
- **MascotModal（吉祥物弹窗）**：同上，加 `isDarkColor()` 函数独立判断卡片明暗
- **DropdownMenu（左上角菜单）**：硬编码 `text-white/70` → 基于 `t.bg` 明暗计算
- **DateNavigator（日历日期导航）**：硬编码 `text-white/85` → `t.cardText`
- **Relations custom-select**：option 硬编码 `#1A1A1A` / `#F5F1EB` → theme-aware
- **Settings 下拉菜单**：`<option>` 加显式背景+文字色
